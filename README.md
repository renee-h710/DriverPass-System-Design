# DriverPass-System-Design


## **Project Overview**

DriverPass is a system designed for a real client, **DriverPass**, a company that helps students prepare for their DMV driving tests. The client wanted a **web‑based system** that allows students to take online classes, complete practice tests, and schedule driving lessons. The system needed to support multiple user roles: students, secretaries, drivers, the owner, and IT administrators, each with their own access levels. The overall goal was to improve student readiness by offering lessons, assessments, driver feedback, and up‑to‑date DMV rule changes all in one platform. 
***

## **What I Did Well**

I did particularly well in creating a **clear and organized breakdown of system requirements**. Both functional and nonfunctional requirements were documented thoroughly, including lesson scheduling rules, package management, user roles, security expectations, and platform needs. I also developed detailed **UML diagrams** that accurately model how each component of the system interacts, helping to establish a strong foundation for the technical design. 
***

## **What I Would Revise**

If I were to revise one part of the work, I would improve the **limitations section** of the Business Requirements Document. While the content explains time, budget, and resource constraints, I would expand it to more clearly describe how these constraints affect the system’s initial scope, especially the decision to launch a **browser‑only MVP**. Adding more direct connections between limitations and design trade‑offs would make the documentation stronger. 
***

## **Interpreting and Implementing User Needs**

To interpret the user’s needs, I relied heavily on the scenario details, focusing on what each user role must accomplish. These needs were translated into system behaviors such as:

*   Creating comprehensive student profiles
*   Scheduling 2‑hour driving lessons
*   Providing online classes and assessments
*   Tracking driver notes, test results, and progress
*   Allowing the owner and IT admin to manage packages and reports
*   Incorporating DMV updates into learning content  
    Each requirement directly ties back to the problems DriverPass wanted to solve: lack of driving experience, road‑rule knowledge, and limited visibility into student progress.

***

## **My Approach to Software Design**

My approach to designing software is based on **progressive refinement**:

1.  Start by identifying the core user needs and constraints.
2.  Translate those needs into clear functional and nonfunctional requirements.
3.  Visualize system behavior through **UML diagrams**, ensuring workflows and interactions make sense.
4.  Define the technical architecture: roles, integrations, database structures, and security controls.
5.  Document assumptions, limitations, and dependencies to avoid hidden risks.

Going forward, I would continue using strategies like:

*   **User‑centric requirement modeling**
*   **Iterative diagramming**
*   **Early identification of constraints**
*   **Modular system design** for easier scaling
*   **Frequent communication loops** with stakeholders to validate assumptions
